# Feature Specification: Restaurant POS App

**Feature Branch**: `001-restaurant-pos`
**Created**: 2026-04-18
**Status**: Draft
**Input**: User description: "음식점에서 주문을 받고 일별로 매출을 정산하는 POS 앱을 개발해줘. 앱의 사용자는 점주이며 손님이 주문하면 점주가 주문을 생성하고 음식이 손님에게 전달되면 전달되었다는 상태를 관리하며 하루 장사가 끝나면 매출을 정산하여 일일 매출 보고서를 볼 수 있도록 해줘. 해당 앱은 좌석 및 메뉴에 대한 설정을 할 수 있어야해."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 주문 접수 및 음식 전달 상태 관리 (Priority: P1)

점주는 손님이 자리에 앉아 주문하면 POS 앱에서 해당 좌석을 선택하고 메뉴를 골라 주문을 생성한다. 주방에서 음식이 완성되어 손님에게 전달되면, 점주는 해당 주문을 "전달 완료"로 표시하여 진행 상태를 추적할 수 있다.

**Why this priority**: POS 앱의 핵심 가치이자 실제 영업이 이루어지는 최소 기능. 이 스토리만 구현되어도 음식점은 기본적인 주문 관리와 서빙 추적이 가능하다.

**Independent Test**: 사전 등록된 좌석·메뉴 데이터가 존재하고 활성(영업 중) 영업일이 개시되어 있는 상태에서, 임의의 좌석을 선택해 메뉴를 담아 주문을 생성한 뒤 주문 목록에서 해당 주문을 "전달 완료"로 전환할 수 있는지 확인한다. 주문 생성 → 준비중 → 전달 완료로 상태가 정확히 전이되고 총액이 바르게 계산되면 성공.

**Acceptance Scenarios**:

1. **Given** 좌석 3번이 비어 있고 메뉴 "김치찌개(₩9,000)"가 판매 중, **When** 점주가 좌석 3번을 선택하고 김치찌개 2인분을 담아 주문을 확정, **Then** 주문이 "준비중" 상태로 생성되고 총액은 ₩18,000으로 표시된다.
2. **Given** "준비중" 상태의 주문이 존재, **When** 점주가 해당 주문을 "전달 완료"로 표시, **Then** 주문 상태가 "전달 완료"로 바뀌고 전달 시각이 기록된다.
3. **Given** 좌석 3번에 이미 활성 주문이 있음, **When** 점주가 동일 좌석에 메뉴를 추가 주문, **Then** 기존 주문에 항목이 추가되거나 새 주문으로 연결되어 동일 좌석의 주문 내역이 통합 조회된다.
4. **Given** 판매 중지(품절) 상태의 메뉴가 존재, **When** 점주가 해당 메뉴를 주문에 담으려 시도, **Then** 시스템이 품절 안내를 표시하고 담기를 차단한다.
5. **Given** "준비중" 상태의 주문, **When** 점주가 주문 항목을 수정하거나 삭제, **Then** 총액이 즉시 재계산되고 변경 이력이 저장된다.

### User Story 2 - 결제 금액 확인 및 결제 완료 처리 (Priority: P2)

음식이 전달된 뒤 손님이 결제를 요청하면, 점주는 해당 주문의 결제 금액(주문 총액)을 확인하여 손님에게 안내하고, 결제가 이루어지면 주문을 "결제 완료" 상태로 전환한다. 결제 완료된 주문만이 당일 확정 매출로 집계된다.

**Why this priority**: 전달 상태 추적(US1)만으로는 실제 매출이 확정되지 않는다. 결제 확인·완료 처리는 POS가 매출 시스템으로 기능하기 위한 필수 연결고리이므로 MVP 직후 가장 우선되는 기능.

**Independent Test**: "전달 완료" 상태의 주문이 존재하는 환경에서 해당 주문의 결제 금액을 조회하고, "결제 완료"로 전환한 뒤 상태·결제 시각이 정확히 기록되는지 확인한다. 결제 완료 후에는 주문 항목 수정이 차단되는지도 검증한다.

**Acceptance Scenarios**:

1. **Given** 총액 ₩18,000의 "전달 완료" 상태 주문, **When** 점주가 해당 주문의 결제 화면을 연다, **Then** 결제 금액 ₩18,000과 주문 항목 내역이 명확히 표시된다.
2. **Given** 결제 금액이 표시된 주문, **When** 점주가 "결제 완료"를 실행, **Then** 주문 상태가 "결제 완료"로 전환되고 결제 시각이 기록된다.
3. **Given** "결제 완료" 상태의 주문, **When** 점주가 해당 주문의 항목을 수정하거나 삭제하려 시도, **Then** 시스템이 수정 불가 안내를 표시하고 변경을 차단한다.
4. **Given** "준비중" 상태의 주문, **When** 점주가 결제 완료 처리를 시도, **Then** 시스템은 "전달 완료 상태에서만 결제 처리 가능" 안내를 표시하고 차단한다.
5. **Given** 좌석에 "전달 완료"·"결제 완료" 주문이 혼재, **When** 점주가 해당 좌석의 결제 화면을 연다, **Then** 미결제 주문 금액의 합과 개별 주문 내역이 동시에 표시된다.

### User Story 3 - 영업 일과 관리 및 일일 매출 보고서 조회 (Priority: P3)

점주는 매장을 여는 시점에 "영업 시작"을 실행하여 영업일을 개시하고, 하루 영업이 끝나면 "영업 마감"을 실행한다. 영업 마감 시점에 시스템은 해당 영업일(영업 시작 시각 ~ 영업 마감 시각) 동안 결제 완료된 주문을 집계하여 일일 매출 보고서를 생성한다. 점주는 현재 보고서와 과거 영업일의 보고서를 함께 조회할 수 있다.

**Why this priority**: 영업 시작/마감은 매출 집계 구간을 명확히 정의해 보고서 신뢰도를 보장하는 핵심 장치다. US1·US2가 선행되어야 집계할 주문 데이터가 생기며, 영업 시작 상태가 아니면 주문 자체가 시작되지 않는다.

**Independent Test**: "영업 시작"을 실행해 영업일을 개시한 뒤 US1·US2의 흐름으로 결제 완료 주문 몇 건을 생성한다. 이후 "영업 마감"을 실행했을 때 해당 영업 구간의 주문만 집계된 일일 매출 보고서가 정확히 생성되는지 확인한다. 이전 영업일(마감된 상태)의 보고서도 같은 형식으로 조회 가능한지 검증한다.

**Acceptance Scenarios**:

1. **Given** 활성 영업일이 없는 상태, **When** 점주가 "영업 시작"을 실행, **Then** 새 영업일이 "영업 중" 상태로 개시되고 영업 시작 시각이 기록된다.
2. **Given** 이미 "영업 중"인 영업일이 존재, **When** 점주가 "영업 시작"을 다시 시도, **Then** 시스템은 "이미 영업 중" 안내를 표시하고 중복 시작을 차단한다.
3. **Given** 활성 영업일이 없는 상태, **When** 점주가 주문을 생성하려 시도, **Then** 시스템은 "영업 시작 후 주문 가능" 안내를 표시하고 주문 생성을 차단한다.
4. **Given** "영업 중" 상태에서 결제 완료된 주문 10건, 전달 완료·미결제 2건, 취소 1건이 존재, **When** 점주가 "영업 마감"을 실행, **Then** 영업 시작 시각부터 마감 시각까지의 결제 완료 10건 기준으로 총매출·주문 건수·메뉴별 판매 수량·시간대별 분포가 집계된 보고서가 생성되고 영업일이 "마감됨" 상태가 된다.
5. **Given** "전달 완료"이지만 미결제인 주문이 남아 있음, **When** 점주가 영업 마감을 시도, **Then** 시스템은 미결제 주문을 경고로 알리고 처리 방식(강제 결제 완료 기록/다음 영업일 이월/취소)을 확인받는다.
6. **Given** 영업 마감이 완료된 영업일, **When** 점주가 해당 영업일의 보고서를 다시 열람, **Then** 확정된 보고서가 변경 없이 동일하게 조회된다.
7. **Given** 지난 7일간의 마감된 영업일 보고서가 존재, **When** 점주가 기간 필터로 범위를 선택, **Then** 선택 범위 내 영업일별 매출 합계와 베스트셀러 메뉴 순위가 표시된다.
8. **Given** 이전 영업일이 "마감됨" 상태, **When** 점주가 새 영업을 시작, **Then** 독립된 새 영업일이 개시되어 이전 영업일의 보고서와 분리된다.

### User Story 4 - 메뉴 및 좌석 설정 관리 (Priority: P4)

점주는 영업 시작 전 또는 필요 시 언제든 메뉴(이름·가격·카테고리·판매 가능 여부)와 좌석(번호·수용 인원)을 등록·수정·삭제하여 매장 상황에 맞는 환경을 유지한다.

**Why this priority**: US1·US2·US3이 동작하려면 기반 데이터가 필요하지만, 초기에는 시드 데이터로 대체 가능하다. 점주가 직접 CRUD UI를 사용하는 것은 운영 단계에서 중요하므로 가장 낮은 우선순위로 배치.

**Independent Test**: 설정 화면에서 메뉴와 좌석을 등록·수정·삭제할 수 있는지 확인하고, 등록된 데이터가 주문 화면(US1)에서 즉시 사용 가능한지 검증한다.

**Acceptance Scenarios**:

1. **Given** 설정 화면, **When** 점주가 신규 메뉴 "제육볶음, ₩10,000, 한식" 등록, **Then** 메뉴 목록과 주문 화면의 선택 가능 메뉴에 즉시 반영된다.
2. **Given** 기존 메뉴가 활성 주문에 포함되어 있음, **When** 점주가 해당 메뉴를 삭제 시도, **Then** 시스템은 활성 주문 참조를 경고하고 판매 중지(soft disable)만 허용한다.
3. **Given** 좌석 5번이 기존에 등록, **When** 점주가 좌석 5번의 수용 인원을 수정, **Then** 변경 사항이 저장되고 진행 중 주문의 좌석 정보에 영향을 주지 않는다.
4. **Given** 좌석에 활성 주문이 없음, **When** 점주가 해당 좌석을 삭제, **Then** 좌석이 목록에서 제거된다.

---

### Edge Cases

- 영업 마감 처리 후 늦은 주문이 들어오는 경우: 자동으로 다음 영업일의 주문으로 귀속되는지, 별도 경고가 필요한지.
- 동일 좌석에 여러 주문이 혼재할 때 총액/정산 기준.
- 네트워크/기기 장애로 주문 저장 실패 시 데이터 손실 방지 (재시도/로컬 큐).
- 주문 취소 또는 환불 시 당일 매출에서 차감되는 방식.
- 심야 영업처럼 자정을 넘기는 주문을 하나의 영업일로 간주해야 하는 경우.
- 메뉴 가격이 변경된 뒤에도 진행 중 주문에는 기존 가격이 적용되어야 함.
- 좌석 없이 포장(take-out) 주문이 필요한 경우의 처리.
- 동일 영업일의 영업 마감을 실수로 중복 실행하려는 시도.
- 영업 시작 없이 주문을 시도하거나, 이미 영업 중인 상태에서 영업 시작을 재실행하는 경우의 안내/차단.
- 하루에 두 번 영업하는 매장(점심 / 저녁 분리 운영)에서 영업 시작-마감 페어를 복수로 운영해야 하는 경우 — 1차 범위는 단일 영업 구간만 지원.
- 영업 마감 직후 동일 일자에 다시 영업 시작을 하는 경우 — 새 영업일로 분리 개시.
- "결제 완료" 처리 후 손님이 환불을 요청하는 경우: 매출 집계에서 차감 및 취소 이력 기록.
- 점주가 실수로 "결제 완료" 상태로 전환한 뒤 되돌리려는 경우 (결제 완료 해제 가능 여부).
- 분할 결제(일부 손님 현금, 일부 카드) 요청 처리 — 1차 범위에서는 단일 결제로만 처리.
- 결제 완료 없이 손님이 떠난 경우(무전취식 등)에 대한 상태 표시.

## Requirements *(mandatory)*

### Functional Requirements

**메뉴·좌석 설정**

- **FR-001**: 시스템은 점주가 메뉴 항목(이름, 가격, 카테고리, 판매 가능 여부)을 등록·수정·삭제할 수 있도록 제공해야 한다.
- **FR-002**: 시스템은 점주가 좌석(좌석 번호, 수용 인원)을 등록·수정·삭제할 수 있도록 제공해야 한다.
- **FR-003**: 시스템은 활성 주문에서 참조 중인 메뉴·좌석의 삭제를 차단하거나, 판매 중지(soft disable)로 대체할 수 있어야 한다.
- **FR-004**: 시스템은 메뉴 가격 변경 이력을 보관하고, 이미 생성된 주문에는 주문 시점의 가격을 유지해야 한다.

**주문 생성 및 상태 관리**

- **FR-005**: 점주는 좌석을 선택하고 하나 이상의 메뉴 항목(수량 포함)을 담아 주문을 생성할 수 있어야 한다.
- **FR-006**: 시스템은 주문 상태 전이 "준비중 → 전달 완료 → 결제 완료"의 순방향 전이와 "취소됨"(준비중 또는 전달 완료 상태에서만 가능)의 분기 전이를 지원해야 한다.
- **FR-007**: 시스템은 상태 전이 시각(생성, 전달 완료, 결제 완료, 취소)을 모두 기록해야 한다.
- **FR-008**: 시스템은 주문 총액을 각 메뉴의 단가와 수량을 기반으로 자동 계산해야 한다.
- **FR-009**: 점주는 "준비중" 상태의 주문에 한해 항목을 추가·삭제·수정할 수 있어야 하며, 변경 시 총액이 즉시 재계산되어야 한다. "전달 완료" 이후 상태에서는 수정이 차단되어야 한다.
- **FR-010**: 시스템은 품절(판매 중지) 상태 메뉴가 신규 주문 항목으로 담기는 것을 차단해야 한다.
- **FR-011**: 시스템은 좌석별 활성 주문(준비중/전달 완료/미결제) 및 상태를 조회할 수 있도록 제공해야 한다.

**결제 처리**

- **FR-012**: 시스템은 점주가 "전달 완료" 상태의 주문에 대해 결제 화면을 열고 결제 금액(주문 총액) 및 주문 항목을 확인할 수 있도록 해야 한다.
- **FR-013**: 점주는 "전달 완료" 상태의 주문을 "결제 완료" 상태로 전환할 수 있어야 하며, 시스템은 그 외 상태(준비중·취소됨·이미 결제 완료)에서의 결제 완료 처리를 차단해야 한다.
- **FR-014**: 시스템은 좌석별 미결제(전달 완료·미결제) 주문 금액의 합을 조회하여 한 화면에서 여러 주문을 함께 결제 처리할 수 있도록 제공해야 한다.
- **FR-015**: 시스템은 "결제 완료" 이후의 환불 처리 경로를 제공하며, 환불 시 해당 주문을 "환불됨" 상태로 기록하고 매출에서 차감해야 한다.

**영업 일과 관리 및 일일 매출 보고서**

- **FR-016**: 시스템은 점주가 "영업 시작"을 실행하여 새 영업일을 개시하고 영업 시작 시각을 기록할 수 있도록 제공해야 한다.
- **FR-017**: 시스템은 "영업 중" 상태의 영업일이 이미 존재하는 경우 영업 시작 재실행을 차단해야 한다.
- **FR-018**: 시스템은 활성 영업일(영업 중 상태)이 없는 경우 신규 주문 생성 및 주문 상태 변경을 차단해야 한다.
- **FR-019**: 시스템은 점주가 "영업 마감"을 실행하여 활성 영업일을 종료하고 영업 마감 시각을 기록할 수 있도록 제공해야 한다.
- **FR-020**: 시스템은 영업 마감 시 미전달(준비중) 또는 미결제(전달 완료) 주문 존재 여부를 점주에게 경고하고 처리 방식을 선택받아야 한다.
- **FR-021**: 시스템은 영업 마감 시점에 해당 영업일의 영업 시작 시각부터 영업 마감 시각까지의 결제 완료 주문을 집계하여 일일 매출 보고서를 자동 생성해야 한다.
- **FR-022**: 시스템은 일일 매출 보고서에 결제 완료 주문 기준의 총매출, 주문 건수, 취소·환불 건수, 메뉴별 판매 수량, 시간대별 매출 분포, 영업 시작/마감 시각을 포함해야 한다.
- **FR-023**: 시스템은 과거 영업일의 일일 매출 보고서를 읽기 전용으로 조회할 수 있도록 해야 한다.
- **FR-024**: 시스템은 지정된 기간(주/월)의 영업일별 매출 합계 및 베스트셀러 메뉴를 집계해 제공해야 한다.
- **FR-025**: 취소·환불 주문은 해당 영업일 매출에서 차감되어 보고서에 반영되어야 한다.

**비기능/운영**

- **FR-026**: 시스템은 주문 생성·수정·상태 전이(결제 완료·환불 포함) 및 영업 시작·마감 내역을 감사 가능한 형태로 기록해야 한다.
- **FR-027**: 시스템은 단일 기기 기준 점주 1인이 안정적으로 사용 가능해야 하며, 인증은 기기 단위 접근 통제로 충분하다.
- **FR-028**: 시스템은 네트워크 장애 시에도 주문 생성·상태 변경·결제 완료 처리·영업 시작/마감이 가능해야 하며, 연결 복구 시 데이터 정합성이 보장되어야 한다.

### Key Entities

- **메뉴 (Menu Item)**: 판매 가능한 음식 항목. 이름, 가격, 카테고리, 판매 가능 여부, 생성·수정 이력을 가진다.
- **좌석 (Seat)**: 매장 내 손님이 앉는 물리적 위치. 좌석 번호, 수용 인원, 활성 주문과의 연결을 가진다.
- **주문 (Order)**: 한 좌석에 귀속된 손님의 주문 묶음. 좌석, 상태(준비중/전달 완료/결제 완료/취소됨/환불됨), 생성 시각, 전달 시각, 결제 시각, 취소/환불 시각, 총액, 주문 항목 목록을 가진다.
- **주문 항목 (Order Item)**: 주문에 담긴 개별 메뉴 라인. 메뉴 참조, 수량, 주문 시점 단가, 소계를 가진다.
- **영업일 (Business Day)**: 점주의 "영업 시작"부터 "영업 마감"까지의 영업 구간. 상태(영업 중/마감됨), 영업 시작 시각, 영업 마감 시각, 해당 구간에 귀속된 주문들을 가진다. 한 시점에 활성(영업 중) 영업일은 최대 1개만 존재한다.
- **일일 매출 보고서 (Daily Sales Report)**: 마감된 영업일에 대한 집계. 영업 시작/마감 시각, 결제 완료 주문 기준 총매출, 주문 건수, 취소·환불 건수, 메뉴별 판매 수량, 시간대별 분포를 가진다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 점주는 손님 주문을 받아 POS에 입력 완료하는 데 평균 30초 이내, p95 기준 60초 이내에 완료한다.
- **SC-002**: 주문 생성부터 저장·화면 반영까지 p95 기준 2초 이내에 완료된다.
- **SC-003**: 전달 상태 전환 및 결제 완료 전환 조작은 터치 후 300ms 이내에 화면에 반영된다.
- **SC-004**: 결제 화면 진입 시 결제 금액·주문 항목이 500ms 이내에 표시된다.
- **SC-005**: 영업 마감 실행부터 일일 매출 보고서 생성까지 5초 이내에 완료된다.
- **SC-006**: 일일 매출 보고서의 총매출·주문 건수 집계는 실제 결제 완료 주문(환불분 차감 후)과 100% 일치한다.
- **SC-007**: 점주가 앱을 처음 접한 후 별도 교육 없이 10분 이내에 주문 생성·전달 처리·결제 완료·매출 확인의 전 과정을 완료할 수 있다.
- **SC-008**: 최소 100석 규모 음식점의 하루 영업(평균 500건 주문)을 성능 저하 없이 처리한다.
- **SC-009**: 네트워크 장애 상황에서 발생한 주문·결제 상태 변경의 99.9%가 연결 복구 후 중복/누락 없이 서버에 동기화된다.

## Assumptions

- 앱의 단일 사용자 역할은 "점주"이며, 여러 직원이 공유 사용하더라도 동일한 권한으로 작동한다. (별도 역할 기반 접근 통제는 범위 외)
- 외부 결제 게이트웨이/단말(카드 리더기·간편결제 SDK) 통합은 범위 밖이며, 결제 수단 상관없이 점주가 결제 완료 여부를 수동으로 표시하는 방식으로 상태를 관리한다. 이 앱은 매출 기록 용도로만 금액을 보관한다.
- 영업일 구분은 점주의 수동 "영업 시작"과 "영업 마감" 페어로 이루어지며, 자동 자정 롤오버는 1차 범위에서 제외한다. 한 시점에 "영업 중" 영업일은 최대 1개만 존재한다.
- 점심/저녁 등 하루에 여러 차례 영업하는 경우는 각 영업을 별도의 영업일로 분리 개시·마감하는 운영 방식을 가정한다. (단일 영업일 내 세션 분리는 범위 외)
- 매장은 단일 지점 기준이며, 다중 지점 통합 관리는 범위 외이다.
- 취소는 "준비중"·"전달 완료" 상태에서만 가능하며, "결제 완료" 이후에는 환불 경로를 사용한다. 부분 취소·부분 환불·할인 정책은 1차 범위에서 제외한다.
- 분할 결제(카드+현금 혼용 등)는 범위 외이며, 주문 단위의 단일 결제 완료로만 처리한다.
- 포장(take-out) 주문은 별도 좌석(예: "포장" 전용 좌석)을 등록하여 처리한다.
- 앱은 태블릿급 단말(10인치 전후)에서 점주가 직접 사용하는 것을 가정하며, 모바일 소형 화면 최적화는 범위 외이다.
- 기본 통화는 대한민국 원(KRW)이며, 다통화 지원은 범위 외이다.
