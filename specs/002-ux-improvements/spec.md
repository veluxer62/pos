## User Scenarios & Testing *(mandatory)*

### User Story 0 - 주문 항목 표시 버그 수정 (Priority: P0 — 긴급)

점주가 주문 상세 화면을 열면 주문에 담긴 메뉴 항목과 수량, 금액이 정확히 표시되어야 한다.
현재 `order_detail_page.dart:57`에서 `items: const []` 하드코딩으로 항목이 전혀 표시되지 않는다.

**Why this priority**: 주문 항목 미표시는 POS 앱의 핵심 기능을 완전히 차단하는 Critical Bug이다. 점주가 무엇을 주문했는지 확인할 수 없어 즉시 수정이 필요하다.

**Independent Test**: 주문을 생성한 뒤 해당 주문의 상세 화면을 열었을 때 주문 항목 목록이 DB에서 조회되어 올바르게 화면에 표시되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 김치찌개 2인분·된장찌개 1인분이 포함된 주문이 생성되어 있음, **When** 점주가 해당 주문의 상세 화면을 열면, **Then** 김치찌개 ×2·된장찌개 ×1 항목과 각 소계, 총액이 화면에 표시된다.
2. **Given** 항목이 1개인 주문이 존재, **When** 주문 상세 화면을 열면, **Then** 해당 단일 항목이 표시되고 빈 목록이 나타나지 않는다.
3. **Given** 여러 주문이 존재하는 환경, **When** 각 주문의 상세 화면을 순서대로 열면, **Then** 각 주문에 속한 항목만 각각 올바르게 표시된다.

### User Story 1 - 60대 접근성 개선 (Priority: P1)

60대 점주가 태블릿 화면에서 메뉴명·금액·상태를 한눈에 읽고, 삭제·취소 동작을 실수 없이 수행할 수 있어야 한다. 시스템 오류 발생 시 기술적 메시지 대신 한국어 안내와 해결 방법이 표시되어야 한다.

**Why this priority**: 실제 사용자인 60대 여성 점주가 작은 텍스트와 숨겨진 인터랙션(롱프레스)으로 인해 앱을 제대로 사용하지 못하는 상황은 사용성 핵심 문제이다.

**Independent Test**: 18sp 이상 폰트로 메뉴명·금액이 표시되는지, 삭제 기능이 명시적 버튼으로 노출되는지, 에러 발생 시 한국어 안내가 나타나는지 각각 위젯 테스트로 검증한다.

**Acceptance Scenarios**:

1. **Given** 주문 화면이 열려 있음, **When** 점주가 메뉴 항목 목록을 보면, **Then** 메뉴명·수량은 최소 18sp, 금액은 최소 20sp bold로 표시된다.
2. **Given** 메뉴 설정 화면에서 메뉴 항목 목록이 표시됨, **When** 점주가 삭제를 원하면, **Then** 각 항목에 명시적 삭제 버튼(또는 슬라이드 삭제)이 표시되고 롱프레스 없이 삭제를 수행할 수 있다.
3. **Given** 점주가 주문 취소 버튼을 탭함, **When** 확인 다이얼로그가 표시되면, **Then** "취소하면 되돌릴 수 없습니다" 경고 문구가 포함된 다이얼로그가 나타나고 확인/취소 버튼이 48dp 이상으로 표시된다.
4. **Given** 영업일이 시작되지 않은 상태에서 주문 생성을 시도함, **When** 에러가 발생하면, **Then** "영업을 시작한 후 주문을 생성할 수 있습니다. 홈 화면에서 영업 시작 버튼을 눌러주세요." 형태의 한국어 안내 메시지가 표시된다.
5. **Given** DB 오류 등 예상치 못한 에러 발생, **When** 화면에 에러가 표시되면, **Then** exception stack trace 대신 "오류가 발생했습니다. 앱을 다시 시작해 주세요." 형태의 한국어 메시지가 표시된다.

### User Story 2 - PENDING 주문 항목 수정 기능 (Priority: P2)

점주는 "준비중(PENDING)" 상태의 주문에서 항목을 추가하거나 삭제할 수 있어야 한다. 품절 메뉴는 주문 화면에서 시각적으로 차단되어 탭 시 안내가 표시된다.

**Why this priority**: FR-009(기존 스펙)에 이미 명시된 요구사항이나 구현이 누락되어 있다. 실제 영업 중 주문 변경은 필수 시나리오이다.

**Independent Test**: PENDING 상태 주문에서 항목 추가·삭제 후 총액이 즉시 재계산되는지 확인한다. DELIVERED/PAID 상태에서는 수정이 차단되는지 검증한다. 품절 메뉴 탭 시 안내 메시지가 표시되는지 위젯 테스트로 확인한다.

**Acceptance Scenarios**:

1. **Given** 김치찌개 1인분이 담긴 PENDING 주문이 존재, **When** 점주가 주문 상세 화면에서 된장찌개 1인분을 추가하면, **Then** 주문에 된장찌개가 추가되고 총액이 즉시 재계산되어 표시된다.
2. **Given** PENDING 주문에 김치찌개·된장찌개·순두부찌개가 담겨 있음, **When** 점주가 된장찌개 항목 삭제 버튼을 탭하면, **Then** 된장찌개가 주문에서 제거되고 총액이 즉시 재계산된다.
3. **Given** DELIVERED 상태의 주문이 존재, **When** 점주가 주문 상세 화면을 열면, **Then** 항목 추가·삭제 버튼이 비활성화(또는 숨김)되어 있고 수정 불가 안내가 표시된다.
4. **Given** "낙지볶음" 메뉴가 품절(isAvailable=false) 상태, **When** 점주가 주문 생성 화면에서 낙지볶음을 탭하면, **Then** 낙지볶음이 흐리게 표시되어 있고 "현재 판매하지 않는 메뉴입니다." 안내가 나타나며 담기가 차단된다.
5. **Given** PENDING 주문에서 마지막 항목 1개만 남은 상태, **When** 점주가 해당 항목을 삭제하려 시도하면, **Then** "주문 항목이 최소 1개 이상이어야 합니다." 안내가 표시되고 삭제가 차단된다.

### User Story 3 - 외상 계좌 데이터 강화 (Priority: P3)

점주는 외상 계좌에 연락처(전화번호)와 메모를 저장할 수 있어야 한다. 외상 거래 이력에서 해당 주문의 상세 화면으로 바로 이동할 수 있어야 한다.

**Why this priority**: 음식점에서 단골 손님의 연락처는 외상 대금 회수에 필수 정보이다. 거래 이력에서 주문 확인도 자주 발생하는 시나리오이다.

**Independent Test**: CreditAccount 등록 시 phone/note 필드가 저장·조회되는지 DB 테스트로 확인한다. 기존 데이터(phone/note 없음)가 마이그레이션 후에도 정상 조회되는지 검증한다. 거래 이력에서 주문 상세로의 네비게이션이 동작하는지 위젯 테스트로 확인한다.

**Acceptance Scenarios**:

1. **Given** 외상 계좌 등록 화면, **When** 점주가 고객명 "김단골", 연락처 "010-1234-5678", 메모 "매주 월요일 방문"을 입력하고 저장하면, **Then** 계좌가 생성되고 연락처·메모가 저장된다.
2. **Given** phone/note 없이 기존에 등록된 외상 계좌가 존재, **When** 앱을 업데이트(마이그레이션)한 후 해당 계좌를 조회하면, **Then** 기존 계좌가 정상 표시되고 phone/note 필드는 빈 값으로 표시된다.
3. **Given** "김단골" 외상 계좌의 거래 이력 화면에 외상 발생 이력이 표시됨, **When** 점주가 해당 이력 항목을 탭하면, **Then** 연결된 주문의 상세 화면으로 이동한다.
4. **Given** 외상 계좌 상세 화면, **When** 점주가 수정 버튼을 탭하여 연락처를 변경하면, **Then** 변경된 연락처가 저장되고 계좌 상세 화면에 반영된다.
5. **Given** 외상 계좌 목록 화면, **When** 점주가 잔액이 있는 계좌를 탭하면, **Then** 고객명·연락처·메모·현재 잔액·거래 이력이 함께 표시된다.

### User Story 4 - 성능 및 데이터 백업 개선 (Priority: P4)

좌석 그리드 화면의 N+1 쿼리를 제거하여 초기 로드 성능을 개선한다. 점주가 영업 데이터를 JSON/CSV로 내보내 이메일로 전송할 수 있다.

**Why this priority**: N+1 쿼리는 좌석 수 증가 시 성능 저하를 유발한다. 데이터 백업은 기기 분실·고장 시 매출 데이터 손실을 방지하는 안전망이다.

**Independent Test**: SeatGridPage 초기 로드 시 DB 쿼리 횟수가 1회(batch)인지 확인한다. 내보내기 실행 시 Share sheet가 표시되고 생성된 JSON 파일에 영업 데이터가 포함되어 있는지 검증한다.

**Acceptance Scenarios**:

1. **Given** 10개 좌석이 등록되어 있고 각각 활성 주문이 존재, **When** 좌석 그리드 화면이 로드되면, **Then** DB 쿼리가 1회(batch) 실행되어 모든 좌석의 주문 상태가 표시된다 (N+1 쿼리 미발생).
2. **Given** 좌석 그리드 화면, **When** 10개 좌석의 데이터가 로드될 때, **Then** 화면 렌더링이 300ms 이내에 완료된다.
3. **Given** 영업 데이터가 존재하는 앱, **When** 점주가 설정 화면에서 "데이터 내보내기"를 실행하면, **Then** 영업일·주문·외상 거래 데이터가 포함된 JSON 파일이 생성되고 Share sheet가 표시된다.
4. **Given** 내보내기로 생성된 JSON 파일, **When** 파일 내용을 확인하면, **Then** 모든 영업일 데이터(날짜·총매출·주문 건수)가 포함되어 있다.
5. **Given** 내보내기 실행 중 Share sheet가 열림, **When** 점주가 이메일 앱을 선택하면, **Then** 생성된 JSON 파일이 첨부된 이메일 작성 화면이 열린다.

### Edge Cases

- US2: 주문 항목 추가 시 해당 메뉴가 그 사이에 품절로 변경된 경우 — 추가 차단 및 안내.
- US2: 마지막 항목을 삭제하려 할 때의 최소 1개 항목 강제.
- US3: phone 필드에 숫자 외 문자(하이픈 등) 입력 허용 여부 — 문자열 자유 입력 허용, 형식 검증 없음.
- US3: 거래 이력에서 연결 주문이 이미 삭제된(취소된) 경우 딥링크 처리 — 안내 메시지 표시.
- US4: 내보내기 실행 시 데이터가 없는 경우 — "내보낼 데이터가 없습니다." 안내.
- US4: 배치 쿼리 전환 후 실시간 업데이트(새 주문 생성) 반영 여부 — watch stream 유지.

## Requirements *(mandatory)*

### Functional Requirements

**버그 수정**

- **FR-B01**: 시스템은 주문 상세 화면에서 해당 주문의 항목 목록을 DB에서 조회하여 표시해야 한다. `items: const []` 하드코딩을 제거하고 실제 데이터를 연결한다.

**접근성**

- **FR-A01**: 시스템은 주문·결제·외상 화면의 메뉴명·항목명 텍스트를 최소 18sp로 표시해야 한다.
- **FR-A02**: 시스템은 주문·결제·외상 화면의 금액 텍스트를 최소 20sp bold로 표시해야 한다.
- **FR-A03**: 시스템은 메뉴 설정·좌석 설정 화면에서 각 항목에 명시적 삭제 버튼(아이콘 또는 슬라이드 삭제)을 제공해야 한다. 롱프레스만으로 삭제하는 패턴을 제거한다.
- **FR-A04**: 시스템은 주문 취소 확인 다이얼로그에 "취소하면 되돌릴 수 없습니다." 경고 문구를 포함해야 한다.
- **FR-A05**: 시스템은 exception 메시지를 직접 표시하는 대신, 한국어 안내 메시지와 해결 방법을 제공하는 에러 처리 레이어를 통해 사용자에게 표시해야 한다.

**PENDING 주문 항목 수정**

- **FR-P01**: 시스템은 PENDING 상태 주문에서 점주가 항목을 추가할 수 있도록 `AddOrderItemUseCase`를 제공해야 한다.
- **FR-P02**: 시스템은 PENDING 상태 주문에서 점주가 항목을 삭제할 수 있도록 `RemoveOrderItemUseCase`를 제공해야 한다. 단, 마지막 항목은 삭제할 수 없다.
- **FR-P03**: 항목 추가·삭제 시 주문 총액이 즉시 재계산되어 화면에 반영되어야 한다.
- **FR-P04**: DELIVERED 이후 상태의 주문에서는 항목 추가·삭제 버튼이 비활성화되어야 한다.
- **FR-P05**: 품절(isAvailable=false) 메뉴는 주문 생성 화면에서 흐리게(opacity 0.4) 표시되고 탭 시 "현재 판매하지 않는 메뉴입니다." 안내를 표시하며 담기를 차단해야 한다.

**외상 계좌 데이터 강화**

- **FR-C01**: `CreditAccount` 엔티티에 `phone` (nullable String) 및 `note` (nullable String) 필드를 추가해야 한다.
- **FR-C02**: 시스템은 외상 계좌 등록·수정 화면에서 연락처와 메모를 입력할 수 있는 UI를 제공해야 한다.
- **FR-C03**: 기존 외상 계좌 데이터는 drift 마이그레이션을 통해 phone=null, note=null로 보존되어야 한다.
- **FR-C04**: 외상 거래 이력에서 외상 발생 이력 항목을 탭하면 연결된 주문 상세 화면으로 이동해야 한다.

**성능 및 백업**

- **FR-X01**: `SeatDao`는 모든 좌석과 각 좌석의 활성 주문을 한 번의 JOIN 쿼리(batch)로 조회하는 `watchAllWithActiveOrders()` 메서드를 제공해야 한다.
- **FR-X02**: `SeatGridPage`는 기존 N+1 provider 대신 batch 쿼리 기반 provider를 사용해야 한다.
- **FR-X03**: 시스템은 설정 화면에서 "데이터 내보내기" 기능을 제공하며, 전체 영업일·주문·외상 거래 데이터를 JSON 형식으로 직렬화하여 기기의 Share 기능으로 전송할 수 있어야 한다.
- **FR-X04**: 내보내기 파일명은 `pos_backup_YYYYMMDD.json` 형식을 따른다.

### Key Entities (변경·추가 사항)

- **외상 계좌 (CreditAccount)** *(기존 엔티티 확장)*: `phone` (nullable String, 연락처), `note` (nullable String, 메모) 필드 추가.

## Assumptions

- CreditAccount 스키마 변경(phone/note 추가)은 drift `addColumn` 마이그레이션으로 nullable 컬럼 추가하며 기존 데이터에 영향 없다.
- 내보내기는 로컬 임시 파일 생성 후 `share_plus` 패키지를 통해 Share sheet를 호출하는 방식으로 구현한다.
- 배치 쿼리 전환(`SeatGridPage`) 후 실시간 watch stream은 기존과 동일하게 유지된다.
- 에러 메시지 한국어 처리는 presentation 레이어의 유틸 함수로 구현하며, domain 레이어 exception 클래스는 변경하지 않는다.
- `AddOrderItemUseCase`와 `RemoveOrderItemUseCase`는 기존 `CreateOrderUseCase` 패턴을 따른다.
